# Phase 3 — Visual Knowledge Mapping

> **Feature flag**: `knowledge_map_enabled`
> **Inspired by**: Litmaps, Research Rabbit, Scite
> **Depends on**: Phase 1 (paper metadata feed), Phase 6 (citation edges)
> **Unlocks**: Standalone; enriches Phase 5 (network view of collaborators)
>
> **Existing feature preserved**: `document_map.py` (`GET /<pid>/map`) + `templates/project/document_map.html`
> provides a Document/Code bipartite graph (qualitative analysis — which codes appear in which docs).
> Phase 3 does NOT touch this page. The new Knowledge Map is a **Citation Network** graph
> at the new URL `GET /<pid>/knowledge-map`, serving a completely different use case: quantitative
> literature exploration via paper-to-paper citation edges. Both features coexist and
> link to each other in the project navigation.

---

## 1. Goal

Replace list-based literature navigation with an interactive visual graph where
researchers can see how papers relate to each other, discover missing literature
by following citation chains, and understand the temporal evolution of their
research field — all without leaving the platform.

---

## 2. Features

### 2.1 Citation Network Graph

An interactive force-directed graph showing papers in a project or search result
and the citation edges between them.

**User actions**
- Open the Knowledge Map from any project page or from the global nav.
- See papers as nodes, sized by citation count, coloured by year bracket.
- Hover a node to see: title, authors, year, citation count, is-in-library badge.
- Click a node to open a side panel with abstract and action buttons (save, view).
- Pan and zoom with mouse/touch.
- Toggle edge types: "cites", "co-cited", "co-author".
- Filter by year range slider.
- Highlight the shortest path between any two selected papers.
- Export graph as PNG or SVG.

**System behaviour**
- Graph data built from:
  1. Papers in the user's project library (always present as starting set).
  2. Their direct citations (fetched from Crossref / Semantic Scholar API).
  3. Papers that cite them (fetched from Semantic Scholar "citations" endpoint).
- Edges typed as `cites` (A references B) or derived `co-cited` (A and B both
  cited by C) after local computation.
- Graph computation happens server-side, delivered as a JSON response.
- Client renders with a lightweight graph library (vis-network or sigma.js via CDN,
  no bundler required).
- Large graphs (>500 nodes) are thinned server-side by minimum edge weight.

**Architecture**
- New service: `KnowledgeGraphService` — builds nodes/edges from library + external data.
- New adapter: `SemanticScholarCitationAdapter` — fetches reference/citation lists.
  (Extends Phase 1's SemanticScholarAdapter with citation-fetch methods.)
- New model: `KnowledgeGraphCache` — serialised graph JSON per (user, project, date).
- Route: `GET /projects/<pid>/knowledge-map` — graph page.
- Route: `GET /projects/<pid>/knowledge-map/data` — graph JSON (cached, async build).

### 2.2 Topic Cluster View

Group nodes into topic clusters computed from abstract text, shown as coloured
convex hulls around related groups.

**User actions**
- Toggle "Topic Clusters" overlay on the graph.
- See each cluster labelled with its top 3 keywords.
- Click a cluster label to see the list of papers in it.
- Filter the graph to show only papers in a selected cluster.

**System behaviour**
- Clustering runs server-side using K-Means over the same embeddings used for
  recommendations (Phase 1). Number of clusters auto-selected by silhouette score
  (between 3 and 15).
- Cluster labels generated by extracting top TF-IDF terms per cluster.
- Cluster assignments stored in `KnowledgeGraphCache.clusters_json`.

**Architecture**
- `KnowledgeGraphService.assign_clusters(nodes)` method — returns cluster assignments.
- Cluster computation is a background task; results merged into the cached graph.

### 2.3 Temporal Evolution View

A timeline layout showing how papers and their connections evolved by year,
helping researchers see when a research area emerged, peaked, or shifted.

**User actions**
- Toggle to "Timeline" layout mode.
- Papers arranged in horizontal year lanes.
- Edges still visible, now showing cross-year citation flow.
- Scrub a year-range slider to reveal the graph as it existed at any point in time.

**System behaviour**
- Same graph data as 2.1, rendered with year-based X-axis.
- Nodes without a known year placed in an "Unknown" lane.
- Slider filters nodes/edges by `year <= slider_value`.

**Architecture**
- Timeline rendering is purely client-side from the same graph JSON.
- `KnowledgeNode.year` used for lane placement (see MODEL.md contract).

### 2.4 Paper Discovery via Graph Traversal

Surface papers connected to the current library but not yet saved — presented
as "ghost nodes" you can pull into the library.

**User actions**
- Ghost nodes (papers not in library) shown in a lighter style at the graph periphery.
- Hover a ghost node to see abstract.
- Click "Add to library" to save directly without leaving the graph.
- "Expand" any node to fetch its next-degree connections.

**System behaviour**
- Ghost nodes fetched up to 2 hops from the current library.
- Addition saves to `ReadingListItem` (Phase 1) or directly to project library.
- Limit: max 300 ghost nodes per render to avoid graph overload.

**Architecture**
- `KnowledgeGraphService.expand_node(doi)` — fetches neighbours from Semantic Scholar.
- Route: `POST /projects/<pid>/knowledge-map/add-node` — saves ghost node.

### 2.5 Global Knowledge Map (Cross-Project)

A map spanning all of the user's projects, revealing thematic bridges between
research areas they are working on simultaneously.

**User actions**
- Open Global Map from main nav.
- All project libraries shown in distinct colour schemes.
- Papers shared across projects highlighted.
- Clusters labelled per-project with override option.

**Architecture**
- `KnowledgeGraphService.build_global_graph(user_id)` method.
- Route: `GET /knowledge-map` (no project scope).
- Separate `KnowledgeGraphCache` entry with `project_id = NULL`.

## RAG / Chunk Template Notes

`KnowledgeGraphService` supports two paths. Choose at runtime based on whether
the server has a graph runtime enabled for the project collection.

### Vector path (default fallback)

Uses `system-semantic` template and raw embeddings:
```python
# Check once per project — cache result
ok, profile = beep_ai_client.get_collection_organization_profile(collection_id)
graph_enabled = profile.get("graph_enabled", False)

if not graph_enabled:
    # Vector path: embed all abstracts and run K-Means locally
    ok, vectors = beep_ai_client.get_embeddings(
        texts=[doc["abstract"] for doc in library_docs]
    )
    # ClusteringService.assign_clusters(vectors) → cluster_ids
```

### GraphRAG path (preferred when ≥ 20 papers and graph runtime available)

Uses `system-graph-rag` template. The server's ingestion pipeline runs entity
extraction + Leiden community detection at index time. No local clustering needed.

```python
if graph_enabled:
    # Communities ARE the topic clusters — query via global search mode
    ok, community_data = beep_ai_client.rag_query(
        query="What are the main themes and research communities in this corpus?",
        collection_id=collection_id,
        quality_mode="high",
        # GraphRAG handles global themes natively
    )
    # community_data.results contain community summaries as cluster labels
```

**GraphRAG collection setup** (one-time, done by project admin when enabling knowledge map):
```python
ok, templates = beep_ai_client.list_chunk_templates()
slug_map = {t["slug"]: t["template_id"] for t in templates}

# Create a dedicated graph collection (separate from the evidence collection)
ok, graph_coll = beep_ai_client.create_rag_collection(
    name=f"project_knowledge_graph_{project_id}",
    user_id=str(user_id),
    is_public=False,
    chunk_template_id=slug_map["system-graph-rag"],
)

# Attach a graph extraction profile (entities + relationships + Leiden)
ok, profiles = beep_ai_client.list_graph_extraction_profile_options()
default_profile = next((p for p in profiles if p.get("is_default")), profiles[0])

beep_ai_client.update_collection_organization_profile(
    collection_id=graph_coll["id"],
    organization_profile={"graph_mode": "graphrag"},
    graph_extraction_profile_id=default_profile["profile_id"],
    user_id=str(user_id),
)
```

**Benefit summary:**
- Leiden communities → Topic Cluster View labels (no TF-IDF needed)
- Entity/relationship graph → `KnowledgeEdge` objects typed as `related_concept | co_cited`
- Community summaries → cluster labels in the UI (section 2.2)
- Local `ClusteringService` K-Means is only needed on the vector path fallback

> **Custom templates**: Users with specialised corpora (e.g. clinical trials,
> legal documents) can apply a custom template to the graph collection from
> `Project Settings → Knowledge Map → Chunk Template`. The GraphRAG entity
> extraction config is separately controlled via the graph extraction profile
> (see `update_collection_organization_profile`). See `MODEL.md` section 9.10
> for the chunk template management design.

## 6. UI Design

> See **MODEL.md §10** for global nav, hub architecture, shared components, and the full file inventory.
> This section covers the interaction design unique to Phase 3.

### 6.1 File Pairs

| Template | JS | CSS |
|---|---|---|
| `templates/knowledge_map/knowledge_map.html` | `static/js/knowledge_map/knowledge_map.js` | `static/css/knowledge_map/knowledge_map.css` |
| `templates/knowledge_map/global_map.html` | *(reuses knowledge_map.js)* | *(reuses knowledge_map.css)* |

> Graph library: **sigma.js** (preferred for large graphs) or vis-network, loaded from CDN. No build step.

### 6.2 Knowledge Map Canvas Layout

```
+--[Toolbar]-------------------------------------------------------+
| [Project picker] | [Search node] | [View v] | [Export v]         |
| [Clusters OFF/ON] [Timeline OFF/ON] [Fullscreen]                 |
+--[Canvas]----------------------------------------[SidePanel]-----+
|                                                  |               |
|         (force-directed graph)                   |  Node detail  |
|                                                  |  panel        |
+------------------------------------------------------------------+
| [+ Zoom] [- Zoom] [Reset view]  | Timeline scrubber (when active)|
+------------------------------------------------------------------+
```

**Node types and styling**:
- Library node: solid circle, colour = cluster colour, opacity 1.0.
- Ghost node (not in library): dashed border circle, opacity 0.45.
- Active/selected: ring highlight `var(--focus-ring-color)`, size +20%.

**Edge types**: `cites` = directed solid arrow; `co_cited` = dashed undirected; `related_concept` = dotted.

### 6.3 View Modes

**Default — Citation Graph**: force-directed layout. Nodes = papers. Edges = citation relationships.

**Topic Clusters toggle** (`[Clusters]`):
- When ON: convex hull overlay drawn around each Leiden community.
- Hull colour = cluster palette (6 theme-aware accent colours cycling).
- Cluster legend slides in from left: colour swatch + label + paper count.
- Label = community summary first sentence (<=40 chars) OR TF-IDF top-terms fallback.

**Timeline Mode toggle** (`[Timeline]`):
- When ON: nodes arranged into year-lane columns (oldest left, newest right).
- Year scrubber slider below canvas: drag to filter nodes by publication year range.
- Nodes outside the range faded to opacity 0.15 (not removed — preserves edge context).
- Timeline mode and fullscreen are mutually exclusive (layout clash).

### 6.4 Node Interactions

| Interaction | Behaviour |
|---|---|
| Click node | Opens right-hand SidePanel; content = compact `PaperCard` (reused from Phase 1) |
| Double-click node | Expands — loads 1-hop neighbour ghost nodes via `POST /knowledge-map/expand/<id>` |
| Click ghost node | SidePanel shows paper metadata + **"Add to library"** button (cross-phase to Phase 6 smart import) |
| Right-click node | Context menu: "Synthesise around this paper" / "Open in new tab" / "View Citation Contexts" |
| Click cluster hull | Selects all nodes in cluster; SidePanel shows cluster summary + **"Synthesise this theme"** button |

### 6.5 SidePanel

Shows `PaperCard` (compact). Sections (collapsed by default):
- **Connections**: list of direct neighbours with relationship type badge.
- **Actions**: "Synthesise around this paper" — cross-phase to Phase 2 (MODEL.md §10.4).

### 6.6 Toolbar Actions

- **Export ▾**: PNG (canvas snapshot), SVG (sigma SVG export), JSON (node/edge data).
- **Project settings [cog]**: opens `ChunkTemplatePicker` for the project graph collection (MODEL.md §9.8).

### 6.7 Responsive / Mobile

Below 600 px width: graph canvas is replaced by a flat list of `PaperCard` (same data). A banner note: "Knowledge Map is available on larger screens."

> Graph library (sigma.js / vis-network) loaded from CDN. No build step required.

## 8. Acceptance Criteria

- [ ] Project with ≥ 2 papers renders a visible graph.
- [ ] Citation edges fetched from Semantic Scholar appear within 10 s.
- [ ] Ghost nodes appear for papers that cite library papers but are not in library.
- [ ] "Add to library" from graph adds `ReadingListItem` without page reload.
- [ ] Topic clusters appear as coloured hulls with keyword labels.
- [ ] Timeline layout places papers in correct year lanes.
- [ ] Year slider filters correctly in < 100 ms (client-side).
- [ ] Graph exports as PNG with all visible nodes included.
- [ ] `knowledge_map_enabled=False` returns 404 for all graph routes.
- [ ] Graph with > 500 nodes is thinned server-side to a renderable count.
